# Demo Walkthrough

## Purpose

This walkthrough is designed to help a reviewer understand PureLaka as a **commerce analytics and reporting project** rather than as a generic e-commerce interface.

The best demo flow is to begin with reporting and operational surfaces, then use commerce pages only as supporting context.

---

## 1. Start with the Analytics Dashboard

### What to show
- KPI cards
- chart outputs
- reporting layout
- date-range or snapshot controls
- export-oriented reporting context

### What this proves
- business-facing KPI thinking
- reporting orientation
- visual delivery of transactional data
- analytics value beyond storefront functionality

### Why it comes first
This is the strongest proof that PureLaka is an analytics/reporting product rather than only a commerce UI.

---

## 2. Open the Monitoring Dashboard

### What to show
- issue counts
- check status
- mismatch / data-quality indicators
- review workflow
- run-check controls if applicable

### What this proves
- operational thinking
- data-quality awareness
- control-oriented system design
- production-minded review surfaces

### Why it matters
This is one of the strongest differentiators in the project because it makes the system feel more credible and operationally mature.

---

## 3. Review the Orders List

### What to show
- search and filters
- staff-facing order visibility
- order state reporting
- export-ready reporting context

### What this proves
- transactional workflow understanding
- order-level business process visibility
- practical reporting around real business entities

### Why it matters
This moves the demo from “analytics overview” into “real business workflow handling.”

---

## 4. Review an Order Detail Page

### What to show
- payment context
- refund workflow visibility
- order summary
- audit-related context

### What this proves
- lifecycle depth
- transaction-level understanding
- payment/refund awareness
- event/accountability thinking

### Why it matters
This is where the project demonstrates that it is not only about dashboards, but also about underlying workflow credibility.

---

## 5. Review the Subscriptions / MRR Surface

### What to show
- active vs inactive states
- recurring revenue context
- lifecycle state handling
- churn-related metrics

### What this proves
- subscription commerce relevance
- recurring revenue awareness
- analytics value beyond one-time orders

### Why it matters
This expands the project from order reporting into broader commerce intelligence.

---

## 6. Use Checkout and Catalogue as Supporting Context

### What to show
- secure checkout
- product catalogue
- commerce flow support

### What this proves
- the reporting and monitoring layers sit on top of a complete commerce context
- the project includes realistic operational and user-facing flows

### Why these come later
These pages are useful, but they are not the strongest proof of the project’s main value. They should support the story, not lead it.

---

## Best short demo sequence

If a reviewer only has a few minutes, use this order:

1. Analytics Dashboard
2. Monitoring Dashboard
3. Orders List
4. Order Detail
5. Subscriptions / MRR

That sequence shows the project’s strongest value with the least wasted time.

---

## Key message to leave with the reviewer

PureLaka should be understood as a **transactional data product** that uses commerce workflows as the source layer for:

- KPI reporting
- operational monitoring
- payment and refund visibility
- subscription insight
- reviewer-friendly business-facing outputs
